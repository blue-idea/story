# CI 失败指南 (GitHub Actions)

使用此指南快速将现象映射到可能的原因、后续命令和安全修复方案。

## 快速路径

1. 获取失败步骤的输出：`gh run view <id> --log-failed`
2. 确定它是否属于：
   - 配置/工作流语法错误
   - 权限/机密问题
   - 环境不匹配（工具/运行时）
   - 不稳定的测试 / 外部依赖
   - Action/工具链回归
3. 应用最小化的安全修复，然后重新运行失败的部分：`gh run rerun <id> --failed`

## 常见失败（现象 -> 修复方向）

### 工作流未启动 / 缺失必要的检查

- **现象：** 未创建运行；分支保护提示缺少检查；预期的工作流名称不符。
- **检查：** 工作流名称/作业名称是否与分支保护要求匹配；触发器是否包含目标事件/分支。
- **修复方案：**
  - 将 `name:`（工作流）和作业名称与分支保护预期对齐。
  - 修复 `on:` 过滤器（`push.branches`、`pull_request.branches`、`paths`、`paths-ignore`）。
  - 如果工作流被重命名，请相应更新分支保护规则。

### YAML / 工作流语法错误

- **现象：** "Invalid workflow file"（无效的工作流文件）、"Unexpected value"（非预期值）、"Unrecognized named-value"（无法识别的命名值）。
- **修复方案：**
  - 验证缩进和列表/映射结构（YAML 对空格敏感）。
  - 对于包含 `:` 或 `{`/`}` 且意为字面量的字符串，请加上引号。
  - 确保表达式位于 `${{ ... }}` 中，且不与 shell 插值混用。

### “Resource not accessible by integration” / 权限错误

- **现象：** checkout（检出）、commenting（评论）、releases（发布）、packages（包）或 API 调用失败，返回 403/404 错误。
- **检查：** 运行是否为来自 fork 仓库的 `pull_request`（机密/令牌会受到限制）。
- **修复方案：**
  - 在工作流/作业级别添加最小权限 `permissions:`（例如 `contents: read`）。
  - 对于使用 OIDC 的部署，确保在需要处设置 `id-token: write`。
  - 避免采用会将机密暴露给 fork PR 代码的“修复方案”；更倾向于将特权作业拆分为在受信任分支上执行的 `push` 事件。

### 机密不可用（特别是来自 fork 的 PR）

- **现象：** 缺失环境变量、身份验证失败、“机密未设置”、跳过部署步骤。
- **修复方向：**
  - 对于 fork PR，重新设计工作流，使不受信任的代码不需要机密（仅进行构建/测试）。
  - 将特权步骤限制在受信任的事件/分支之后。
- **高风险：** 切换到 `pull_request_target` 可能会在检出 / 运行 PR 代码时泄露机密。

### Checkout / Git 历史 / 子模块问题

- **现象：** 缺少标签/提交、版本控制工具失败、子模块抓取失败。
- **修复方案：**
  - 当需要完整历史记录时，为 `actions/checkout` 使用 `fetch-depth: 0`。
  - 需要时显式启用子模块 (`submodules: recursive`)。
  - 确保私有子模块的凭据可用（通常需要 PAT；请指出风险）。

### 缓存问题

- **现象：** 依赖项过旧、间歇性构建失败、缓存恢复未命中。
- **修复方案：**
  - 在缓存键中包含锁文件，并在缓存格式更改时更新键。
  - 优先使用包管理器的官方缓存策略；除非必要，避免直接缓存 `node_modules`。
  - 如有疑问，在禁用缓存的情况下测试一次以确认。

### 运行时/工具链不匹配

- **现象：** 在本地运行正常但在 CI 中失败；“unsupported engine”；Java/Node/Python 版本不匹配。
- **修复方案：**
  - 显式固定工具版本（使用 setup actions）并与本地开发版本匹配。
  - 确保工作流使用正确的工作目录和配置文件。
  - 对于单体大仓 (Monorepo)，确认每个作业都正确设置了 `working-directory`。

### 不稳定的测试 / 外部服务

- **现象：** 超时、偶发性失败、速率限制。
- **修复方向：**
  - 稳定测试（设置随机种子、隔离共享状态、在有理由的情况下增加超时时间）。
  - 仅针对明确不稳定且幂等的操作添加重试（并跟踪后续处理）。
  - 考虑隔离真正不稳定的测试，并分配明确的工单和负责人。
