# STEP 4 · 规格审查与分析（Analysis）

> **前置**：`design.md`、`data.md`、`api.md` 均已定稿
> **输出文件**：`docs/spec/analysis-report-{{DATE}}.md`（审查报告，归档用）
> **模板参考**：`templates/tpl_analysis_report.md`

---

## 目标

在进入任务拆分前，对所有核心规格文档进行**全面审查**，识别：
- 需求层的歧义与缺失
- 跨文档的不一致与逻辑冲突

所有审查发现**必须记录在审查报告中**，更新规格文档后同步写入变更清单，确保可追溯。

---

## 审查范围

必须逐一审查以下文件：

| 文件 | 审查重点 |
|------|---------|
| `docs/spec/requirements.md` | EARS 完整性、AC 可测试性、expected 字段与 test_type 匹配 |
| `docs/spec/design.md` | 架构与需求的一致性、测试工具选型是否越界写了策略细节 |
| `docs/spec/data.md` | 数据模型与需求实体的映射 |
| `docs/spec/api.md` | 接口契约与 AC 期望值的匹配 |

---

## 审查维度

| 维度 | 检查内容 |
|------|---------|
| 术语一致性 | 是否存在未定义或含义模糊的业务术语？ |
| 用户角色清晰度 | 是否明确谁在使用？权限边界是否清晰？ |
| 触发条件完整性 | EARS 中的 "When" 是否覆盖所有关键场景（含异常、边界）？ |
| 系统响应可验证性 | 验收标准是否可被测试？是否含主观表述？ |
| 前置条件与约束 | 是否遗漏环境、状态或外部依赖前提（如"用户已登录"）？ |
| 跨文档冲突 | 需求、设计、数据、接口之间是否存在逻辑矛盾？ |
| 非功能性需求缺失 | 是否缺少性能、安全、兼容性、可访问性等要求？ |

---

## 澄清问题执行规则

1. 澄清总量**不超过 7 条**，突出 P0、P1 必须澄清的重点问题。
2. **逐条抛出**，不得一次性提出多个问题。
3. 每条必须关联具体需求编号（如 `REQ-001`）。
4. 以 **A / B / C / D 选项**形式呈现，便于快速决策：

```
1. 用户未登录时访问受保护页面，系统应如何响应？【REQ-002】
   A. 直接跳转登录页，不保留当前路径
   B. 跳转登录页，登录后返回原路径
   C. 弹出登录弹窗，不跳转页面
   D. 其他（请说明）
```

5. 用户逐一回复后，据此同步更新相关规格文档，**并在审查报告中记录澄清内容和处理动作**。

---

## 审查报告输出规范

审查完成后，加载 `templates/tpl_analysis_report.md`，输出 `docs/spec/analysis-report-{{DATE}}.md`：

- 每个发现问题逐行列入「问题列表」
- 每条澄清问题及用户回复逐条写入「澄清问题记录」
- 所有对规格文档的修改写入「文档变更清单」
- 填写最终「结论」（可进入 STEP 5 / 待澄清 / 需重大修订）

> 审查报告一旦归档不再修改。若 STEP 5/6 期间发现新问题，须另起新报告记录。
