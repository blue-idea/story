> **加载时机**：所有步骤均默认继承本文件规范，无需重复说明。
> **输出文件**：无（本文件为行为约束，不产出项目文件）

## 沟通规范

1. 所有内容使用**中文**交流，包括代码注释和测试描述。
2. 仅以下内容使用英文：用户界面文案、错误提示消息、日志消息、第三方 API 字段名。
3. 遇到不清楚、歧义、缺失内容时，立即向用户提问，不得自行假设。
4. 表达清晰、简洁、技术准确。
5. 当前任务完成后，提出下一步执行建议。

### 代码语言规范速查

| 内容 | 语言 | 示例 |
|------|------|------|
| 代码注释 | 中文 | `// 计算折扣后的最终价格` |
| 测试描述（`test()`/`it()`/`describe()`） | 中文 | `test('提交空邮箱时返回错误', ...)` |
| 函数名 / 变量名 | 英文（代码惯例） | `calculateDiscount()` |
| 错误消息字符串 | 英文 | `'Email is required'` |
| UI 文案字符串 | 英文 | `'Submit'` |
| Git commit message | 中文或英文均可，项目内保持一致 | — |

## 开发工程化规范

1. **严禁重复**：任何重复逻辑必须抽象为可复用单元（函数、类等）。
2. **检查现有代码**：先分析项目中是否已有类似实现。
3. **模块化设计**：将功能拆分为独立、可复用的模块。
4. **统一接口**：使用一致的参数格式和返回值结构。
5. **配置集中管理**：所有常量和配置项统一放在 `config/` 目录。
6. **优先封装**：将相关功能和数据封装在职责明确的模块中。
7. **环境变量**：使用 `.env` 文件，禁止硬编码密钥或敏感信息。

> **完整协议见**：`phases/tdd.md`（STEP 6 执行时必读）

核心铁律（详细说明和所有例外情况见主文档）：

```
⛔ 没有失败的测试，禁止写任何生产代码
```

顺序：红（写失败测试并亲眼验证）→ 绿（最小实现）→ 重构（保持绿）。

## QA 测试执行规范

### 禁止伪造测试结果

> ⛔ 严禁在未执行真实调用、未获取实际返回值的情况下输出 `PASS`。

- 无法执行的用例须标记 `BLOCKED` 并说明原因。
- 不得跳过、伪造、捏造任何测试结论。

### 测试工具选择策略

| 场景 | 工具 | 备注 |
|------|------|------|
| 单元测试 / 组件测试 | Vitest + Testing Library | — |
| Web UI & E2E 测试 | Playwright MCP | 需截图留证 |
| 移动端 E2E 测试 | Maestro CLI | 需截图留证 |
| API 接口测试 | `curl` | 直接执行，展示实际返回值 |

### 测试数据来源

测试账号与测试数据统一从 `docs/spec/info.md` 获取，**禁止自行捏造**。
