## 阶段 2 · 执行中：严格 TDD 循环

> **主文档**：`phases/tdd.md`（必读，本节为 SDD 集成要点摘要）
> **反模式防护**：添加 Mock 或测试工具时加载 `phases/testing-anti-patterns.md`

### TDD 在不同测试类型下的适配规则

> `_测试类型:` 决定 TDD 铁律如何落地。Unit / API 完全适用，E2E 有专属变体。

| 测试类型 | TDD 适用方式 | "RED" 的形态 |
|---------|------------|------------|
| `Unit` | 严格 TDD，无例外 | 断言失败（feature missing） |
| `API` | 严格 TDD，无例外 | HTTP 状态码 / 响应体不符 |
| `E2E` | **先写骨架，后实现**（见下方规则） | 页面/元素未找到（404 / element not found） |
| `Performance` | 实现后补基准测试，不适用 TDD | — |
| `Security` | 实现后执行检查清单，不适用 TDD | — |

**E2E 的 TDD 变体（必须遵守）**：

E2E 测试的选择器依赖 UI 元素，无法在 UI 存在之前写出完整断言。但铁律的核心精神——**先定义期望行为，再实现**——仍然适用，执行方式调整为：

```
STEP A（RED）：写 E2E 测试骨架
  · 写出完整的用户旅程流程（导航路径、操作步骤、最终断言）
  · 使用语义化选择器（getByRole / getByText），不用 CSS 类
  · 运行测试，确认失败原因是"页面不存在"或"元素未找到"
  · ⛔ 如果测试直接通过，说明测试了已有行为，须修改测试目标

STEP B（GREEN）：实现 UI 功能
  · 按照 E2E 测试骨架描述的旅程来实现，而不是先实现再对齐测试

STEP C（验证）：运行完整 E2E 套件
  · 测试通过，截图存档
  · 不稳定（flaky）→ 查阅 qa_engine.md §第4阶段「不稳定测试分类排查」
```

> **关键原则**：E2E 测试骨架就是"规格"，UI 实现要对齐骨架，而非骨架对齐实现。

---

### 铁律（Iron Law）

```
⛔ 没有失败的测试，禁止写任何生产代码
```

- 先写测试，后写实现——顺序错误则**删除已写代码，从测试重新开始**
- 不得以"参考用途"保留实现代码，删除即删除
- 不得以"已手动测试""太简单""这次特殊"为由跳过

发现自己在找理由跳过？查阅 `phases/tdd.md` §Common Rationalizations。

---

### 与 SDD 规格的绑定规则

每条测试**必须锚定到需求文档**中的 AC，在测试名称或注释中标注：

```typescript
// REQ-002-AC-003：当用户提交空邮箱时，表单 shall 返回 'Email required'
test('提交空邮箱时返回 Email required 错误', async () => {
  // 验证：空邮箱提交应被拒绝
});
```

测试通过后，在代码注释中标注覆盖该 AC 的函数/模块（可追溯性）。

---

### RED — 写失败的测试

按 `phases/tdd.md` §RED 编写：

- 一个测试只测一个行为
- 测试描述格式：`[单元/场景] [条件] [预期行为]`（中文）
- 优先使用真实代码，Mock 只用于不可避免的外部依赖

**必须执行**：
```bash
pnpm test path/to/test.test.ts
```

确认三项后才能进入 GREEN：
1. 测试**失败**（不是报错）
2. 失败信息符合预期
3. 失败原因是功能缺失，不是拼写错误

测试直接通过？说明在测试已有行为，修正测试。

---

### GREEN — 最小实现

写**最简单**能让测试通过的代码，不多写一行：

- 不添加未被测试覆盖的功能
- 不提前重构其他代码
- 不"顺便改进"

**必须执行**：
```bash
pnpm test path/to/test.test.ts
```

测试仍失败？修代码，不改测试。其他测试新增失败？立即修复。

---

### REFACTOR — 清理

仅在全绿后执行：消除重复、改善命名、提取辅助函数。
不得在重构中新增行为，重构过程中持续保持全绿。

---

### Mock 使用前置门控

添加任何 Mock 前，先回答以下问题（参照 `phases/testing-anti-patterns.md`）：

```
1. 这个方法有哪些副作用？
2. 本测试是否依赖这些副作用？
3. 我是否完全理解这个依赖链？

如依赖副作用 → 在更低层级 Mock，而非 Mock 整个高层方法
如不确定     → 先用真实实现跑测试，观察实际需要什么，再 Mock
```

**绝对禁止**（见 `phases/testing-anti-patterns.md`）：
- 断言 Mock 元素的存在（测试 Mock 行为而非真实行为）
- 在生产类中添加仅用于测试的方法
- "为安全起见"随意 Mock

### 本阶段完成核查清单

> 来自 `phases/tdd.md` §Verification Checklist，每项均须 ✅ 才能进入阶段 3

- [ ] 每个新函数 / 方法都有对应测试

**无法全部勾选 = 跳过了 TDD，删除代码重来。**

## 阶段 3 · 执行后：分层 QA 验收

> 根据 TASK 的 `_测试类型:` 字段，执行对应层级的 QA。
> `qa_engine.md` 是本阶段的**主要参考文档**，按需查阅对应章节。

### 3.0 静态检查（所有 TASK 必须通过，任意一项未通过禁止继续）

```bash
pnpm exec tsc --noEmit          # 零 error
pnpm exec eslint src --max-errors=0  # 零 error
pnpm test --run                 # 相关用例 100% 通过
pnpm build                      # 零 error
```

### 3.1 单元测试（test_type: Unit）

> 查阅 `phases/qa_engine.md` §第2阶段

- 遵循 **AAA 模式**（Arrange-Act-Assert）编写测试
- 按命名规范：`[单元] [场景] [预期行为]`
- Mock 规则：**在边界处 Mock，而不是在内部 Mock**
- 覆盖率目标（参照 test_strategy.md 的风险等级）：

  | 风险等级 | 行覆盖率 | 分支覆盖率 | 关键路径 |
  |---------|---------|----------|---------|
  | 内部工具 | ≥70% | ≥60% | 100% |
  | B2B/B2C | ≥85% | ≥80% | 100% |
  | 金融/健康 | ≥95% | ≥90% | 100% |

### 3.2 集成 / API 测试（test_type: API）

> 查阅 `phases/qa_engine.md` §第3阶段

每个端点至少覆盖以下场景（使用 `curl` 执行并保存真实输出）：

```
happy_path    → 201/200 + 响应 Schema 验证 + 数据库记录验证
validation    → 400（缺失字段）/ 422（业务规则违反）
authentication → 401（无 Token）/ 403（权限不足）
edge_cases    → 幂等性 / 特殊字符 / 最大 Payload
error_handling → 503（依赖宕机）/ 429（超速率限制）
```

### 3.3 E2E 测试（test_type: E2E）

> 查阅 `phases/qa_engine.md` §第4阶段

- 识别并覆盖**关键用户旅程**（注册、购买、核心任务等）
- 选择器优先级：`getByRole` > `getByLabelText` > `getByText` > `data-testid` > ❌ CSS 类
- 等待状态而非时间：`waitFor(element)` 而非 `sleep(3000)`
- Web：Playwright MCP，截图存档；移动端：Maestro CLI，截图存档

### 3.4 性能测试（test_type: Performance，按需执行）

> 查阅 `phases/qa_engine.md` §第5阶段

- 先跑 **smoke（冒烟）**验证脚本有效，再跑 **load（负载）**
- 对照 test_strategy.md 中定义的性能预算验收：

  | 指标 | Web 应用 | API |
  |------|---------|-----|
  | p95 响应时间 | <1s | <500ms |
  | 错误率 | <0.1% | <0.1% |

### 3.5 安全测试（test_type: Security，按需执行）

> 查阅 `phases/qa_engine.md` §第6阶段

- 对照 OWASP Top 10 清单逐项检查
- 对所有用户输入使用注入 Payload 测试（SQL / XSS / 路径穿越）
- 验证认证端点的 Rate Limiting 和账户锁定逻辑

### 3.6 输出 AC 验收矩阵

按 `templates/tpl_ac_matrix.md` 格式输出到 `docs/spec/ac/TASK-XXX-AC.md`：

- `PASS`：错误详情填 `—`
- `FAIL`：**必须**填写实际值 vs 期望值差异及可能根因
- `BLOCKED`：**必须**填写阻塞原因及依赖项

## 阶段 3.5 · QA 失败后：自动修复循环

> **触发条件**：阶段 3 的 AC 验收矩阵中存在 `FAIL` 条目时，进入本阶段。
> `BLOCKED` 条目不触发本阶段，保持原状等待依赖解除。
> **所有修复操作在用户确认前禁止执行任何代码变更。**

### 修复循环流程

```
┌─────────────────────────────────────┐
│  AC 矩阵存在 FAIL 条目               │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│  1. 分析阶段（AI 独立完成）           │
│     · 归类每条 FAIL 的根因类型        │
│     · 评估修复影响范围（文件/模块）    │
│     · 判断修复轮次（第几次修复）       │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│  2. 向用户提交修复方案（等待确认）     │
│     · 格式见「修复方案呈现格式」       │
│     · ⛔ 此时禁止修改任何文件         │
└──────────────┬──────────────────────┘
               ↓
        用户三种回应
       ┌────┼────┐
       ↓    ↓    ↓
     确认  调整  跳过
       ↓    ↓    ↓
     执行  修改  标记
     修复  方案  WONTFIX
       ↓         ↓
     重新        进入
     测试        阶段4
       ↓
  更新 AC 矩阵
       ↓
  仍有 FAIL?
  ↙        ↘
是（轮次<3）  否
  ↓            ↓
再次循环      进入阶段4
  ↓
轮次≥3
  ↓
上报用户人工介入
```

---

### 修复方案呈现格式

> AI 在用户确认前，**只输出分析和方案，不执行任何操作**。

```
## 🔧 修复方案（第 N 轮 · 共 X 条 FAIL）

### FAIL-001 · [AC ID: REQ-XXX-AC-YYY]
**根因类型**：逻辑错误 / 类型错误 / 配置缺失 / 环境问题 / 测试数据问题
**根因说明**：[一句话描述实际原因]
**影响范围**：`src/xxx/yyy.ts` 第 N 行 / `xxx` 模块
**修复方案**：
  - [ ] [具体操作1，例如：修改 calculatePrice 函数的边界判断]
  - [ ] [具体操作2]
**预计影响**：仅影响本 AC / 可能影响 [相关 AC 编号]
**风险等级**：低 / 中 / 高

---

### FAIL-002 · [AC ID: REQ-XXX-AC-ZZZ]
...

---
请回复：
  A. 确认全部执行
  B. 调整方案（请说明哪条需要调整）
  C. 跳过某条（请说明 FAIL 编号，标记为 WONTFIX）
```

---

### 根因分类与处理策略

| 根因类型 | 判断依据 | 默认修复策略 |
|---------|---------|------------|
| 逻辑错误 | 实现与 EARS 描述不符 | 修改对应函数/模块 |
| 类型错误 | tsc 报告 / 运行时类型异常 | 修正类型定义或断言 |
| 配置缺失 | 环境变量 / 初始化未完成 | 补充配置，不硬编码 |
| 测试数据问题 | 数据与 `info.md` 不一致 | 对齐 info.md，不捏造数据 |
| 环境问题 | 仅 CI 失败 / 本地通过 | 排查资源限制，添加显式等待 |
| 设计冲突 | 修复需改动接口或数据模型 | **进入 `SKILL.md` §规格回退协议，输出设计冲突通知，等待用户确认后回退** |

> **⚠️ 设计冲突不在修复循环内处理**。若修复方案需要改动 `api.md` / `data.md` / `design.md` 中已定稿的内容，立即停止，按 `SKILL.md` §规格回退协议执行，**不得直接修改规格文件**。

---

### 轮次上限与人工介入

| 修复轮次 | 策略 |
|---------|------|
| 第 1 轮 | 正常修复循环 |
| 第 2 轮 | 在方案中附加「上一轮修复未解决的原因分析」 |
| 第 3 轮 | **必须停止自动修复**，输出完整问题报告请求人工介入 |

**第 3 轮触发时的输出格式**：

```
## ⛔ 已达修复上限，请求人工介入

**任务**：TASK-XXX
**持续 FAIL 的 AC**：[列表]
**已尝试的修复**：
  - 第1轮：[描述]
  - 第2轮：[描述]
  - 第3轮：[描述]
**当前怀疑根因**：[说明]
**建议排查方向**：[说明]
```
