## 澄清问题记录

### 澄清 1 · 用户认证与多租户管理【REQ-001 / REQ-002】

**问题**：在目前的 Web 写作网站中，我们需要在首期版本中实现具体的用户登录，还是仅在本地会话中保存小说项目？

**选项**：
- **A. 个人单机部署版**：不需要登录，所有人访问都展示同一个控制台，直接在数据库中以 `default_user` 保存。
- **B. 轻量级多用户隔离版**：使用 NextAuth.js 或 Supabase Auth 实现邮箱登录/注册，不同用户隔离自己的小说。
- **C. 临时匿名会话版**：不需要登录，但使用随机 UUID 作为 `session_id` 存在 cookie 里，在数据库中进行小说关联，只保证单浏览器隔离。

**用户回复**：**选择 B（一次性开发 NextAuth 轻量级多用户隔离版）**

**处理动作**：更新 `requirements.md` 增加 NextAuth 用户登录需求；更新 `design.md` 架构设计和 `data.md` 数据库设计，合并 NextAuth.js 所需的标准 schemas 表；更新 `api.md` 包含 auth API 路由。

---

### 澄清 2 · 自动写作的故障与终止策略【REQ-004 / REQ-005】

**问题**：如果在 AI 自动写作过程中，外部大语言模型接口连续报错（例如 API 额度不足、超时），或者某一章节连续 3 轮字数/悬念质量校验均失败，系统应当如何向用户呈现？

**选项**：
- **A. 自动暂停并允许重试（推荐）**：自动暂停当前章节的写作，在 UI 界面上将章节状态置为 `failed` 并展示具体的报错日志，同时允许用户点击“手动重试当前章”。
- **B. 忽略错误强制向下**：忽略错误，继续强制生成下一章，只在最后的小说详情页上标记“第 X 章字数未达标”或“第 X 章生成失败”。
- **C. 彻底中断重置**：彻底中断整本小说的写作任务，回滚到 `planning` 大纲确认阶段。

**用户回复**：**选择 A（自动暂停并允许重试）**

**处理动作**：更新 `requirements.md` 的 REQ-005 明确自动暂停和重试按钮的 AC；更新 `design.md` 时序图，体现异常终止和手动重试的连接逻辑；更新 `api.md` 丰富了 SSE 中的 error 消息。

---

### 澄清 3 · 并行写作模式下的上下文传递【REQ-004】

**问题**：在并行写作（subagent-parallel）模式下，由于所有章节同时开始写作，后一章的 AI 无法得知前一章具体生成的内容（只能看到提纲）。这可能会破坏章节结尾“悬念钩子”的连贯性。我们如何处理？

**选项**：
- **A. 正文串行，大纲并行**：并行写作只并行生成“初稿大纲的扩写”，正文的写作仍然必须是串行的。
- **B. 纯并行生成（无上下文）**：AI 写作仅依赖本章大纲提纲和前后章提纲，不考虑前一章生成的正文文字。
- **C. 混合分批并行（推荐）**：分批生成。

**用户回复**：**不支持并行写作模式（只做纯串行自动写作）**

**处理动作**：在 `requirements.md`、`design.md`、`data.md`、`api.md` 中完全移除并行写作（subagent-parallel）相关的配置、参数、大纲拆批计划等描述，系统强制为串行流式写作。
